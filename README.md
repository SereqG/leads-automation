# LeadGen Platform

A lead-generation platform covering the full pipeline: prospect discovery,
data enrichment/analysis, storage, and outbound email campaigns.

The platform is exposed two ways over a shared core:

- a **FastAPI HTTP API** (`backend/main.py`)
- a **CLI tool** built with Typer (`backend/cli.py`)

Both are thin adapters over the same service/selector layer — no business
logic lives in routes or CLI commands themselves.

## How it works

The backend is organized as **vertical slices** under `backend/apps/` — each
feature (prospect discovery, enrichment, campaigns, accounts) owns its own
models, schemas, services, selectors, routes, CLI commands, and tests, rather
than being split across technical layers.

Currently implemented:

### `prospects` slice — prospect discovery

The first working piece of the pipeline: searching for prospects from a list
of Google-style search queries.

- **`backend/data/queries.csv`** — the list of search queries to run, one per
  row (column `google_search_query`).
- **`backend/data/blacklist.txt`** — domains to exclude from search results
  (e.g. marketplaces, social networks, aggregators), grouped under `#`
  comments.
- **`backend/cli.py prospects search`** — validates a search run's inputs
  (`--per-query` count, `--contact-email`, optional `--log-file`), checks
  `queries.csv` for duplicate queries (offering to write a deduplicated
  `queries-copy.csv` if any are found), and logs the whole process to
  `backend/logs/`.

Everything else (enrichment, campaigns, accounts) is scaffolding for future
slices as described in `CLAUDE.md`.

### Shared infrastructure

- **PostgreSQL** — primary datastore, managed via SQLAlchemy 2.0 models and
  Alembic migrations (`backend/migrations/`).
- **Redis + Celery** — broker/backend for async and scheduled jobs (worker +
  beat services), for slice-owned long-running work such as scraping,
  enrichment, and bulk email sending.
- **`backend/core/`** — genuinely cross-cutting code only: the SQLAlchemy
  declarative base, shared exceptions, logging setup.

## Setup

Everything runs in containers via Compose (this project uses **Podman**, not
Docker — the Makefile targets call `podman-compose`).

### Prerequisites

- Podman + `podman-compose`
- `make`

### First-time setup

1. Copy the example env file and adjust values as needed:

   ```bash
   cp .env.example .env
   ```

2. Build the images:

   ```bash
   make build
   ```

3. Start all services (API, PostgreSQL, Redis, Celery worker + beat):

   ```bash
   make up
   ```

4. Apply database migrations:

   ```bash
   make migrate
   ```

The API is then available at `http://localhost:8000`.

### Everyday commands

```bash
make up                     # start all services (detached)
make down                   # stop and remove containers
make build                  # rebuild images (after dependency changes)
make logs                   # tail logs for all services
make logs s=api              # tail logs for one service (api, postgres, redis, worker, beat)

make migrate                 # apply Alembic migrations
make migration m="message"   # generate a new autogenerated migration

make cli cmd="prospects search --per-query 20 --contact-email me@example.com"
                              # run a Typer CLI command inside the api container
make shell-backend            # shell into the api container
make dbshell                  # psql shell into postgres

make test-backend             # run pytest inside the api container
make lint                     # ruff + black --check
```

Always use these `make` targets rather than raw `podman`/`docker` commands —
they keep flags, env files, and service names consistent, and match how CI
runs things.

### Data & logs

`backend/data/` and `backend/logs/` hold local, run-specific files
(`queries.csv`, `blacklist.txt`, dedup copies, log files) and are gitignored
aside from `.gitkeep` placeholders — they aren't shared through the repo.
Populate them yourself, or via the running containers.

## Notes

- Never commit `.env` or any file containing real secrets — only
  `.env.example` (placeholder values) is tracked.
- Anything touching real prospect data or sending real emails is high-risk;
  confirm before running against production data or live email providers.
- See `CLAUDE.md` for the full architectural rules and conventions this
  project follows.
